# 開発運用ガイドライン（セッション/コミット）

本ドキュメントは、**セッションの運用ルール**と**自動コミット**の扱いを整理したものです。

## 1. セッション開始時のルール

- **必ず `docs/AGENTS.md` を参照**してから作業を開始する。
- 以降のやりとりは**日本語で統一**する（英語での回答は行わない）。

## 2. 自動コミットの運用ルール

### 2-1. トリガー
以下のような**肯定的な合図**が来たタイミングで、実装内容を自動コミットする。

- 「OK」「いい感じ」「この方針で進めて」「助かった」などの**肯定的プロンプト**

### 2-2. ステージ範囲

- **作業で変更したものはすべてステージ**してコミットする。
- 不要な変更が混ざっている場合は、必ず整理してからコミットする。

### 2-3. コミットメッセージ

- **日本語で記述**する。
- 例: `公式スポンサーセクションのレイアウト調整`

### 2-4. 実行コマンド

```
npm run commit:auto -- "コミットメッセージ"
```

## 3. 注意事項

- 自動コミットは**セッション内の承認タイミングに合わせて実行**する。
- 変更差分がない場合はコミットを行わない。

## 4. 実装検証と報告のルール（再発防止）

- **断定禁止**: 実行確認が取れていない挙動について「〜になっています」と断定しない。
- **報告の粒度**: 変更内容は「どのファイル/どの設定を変えたか」まで必ず具体化する。
- **確認ステータスを明示**:
  - 「変更済み（未確認）」と「動作確認済み」を必ず区別する。
  - 体感やUI挙動は、確認できていない場合は必ず「未確認」と明記する。
- **検証手順の提示**: 変更の効果を確かめるための可視化手段（例: markers、ログ、ガイド線）を提示する。
- **不一致時の対応**: ユーザー報告と一致しない場合は、即断せず原因候補を列挙し、再現・計測手段を先に提案する。
